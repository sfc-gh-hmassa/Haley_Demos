How the Recommender MCP Actually Works

analyze_csv() + analyze_fft_patterns()
  • Reads CSV with pandas
  • Counts samples, features, missing values
  • Identifies numeric vs categorical columns
  • Computes Pearson correlation per feature vs target (avg_linearity)
  • Tests linear separability with a Perceptron (100 iter convergence)
  • Checks multicollinearity (correlation matrix > 0.8)
  • FFT: computes power spectrum per feature, compares pass vs fail groups
  • Mann-Whitney U test for statistical significance of FFT differences

Output: A characteristics dict with ~12 fields that drive scoring.

score_algorithm() — the scoring engine

Each algorithm has hand-tuned scoring rules based on data characteristics. Example for XGBoost:

  • +30 if samples >= 100
  • +25 if linearity < 0.5 (non-linear)
  • +15 if has categorical features
  • +15 if features > 10
  • +10 if class imbalanced
  • -20 if samples < 50 (overfit risk)
  • +20 if FFT shows strong frequency patterns

~560 lines of scoring logic. Pure Python, no Snowflake dependency. Lifts as-is into our architecture.

Why 1 MCP Per Model Doesn't Scale

All 8 training MCPs do the same thing: load data, preprocess, fit, evaluate, save. The only difference is which model class they instantiate. Isolating each into a separate server was a workaround for dependency conflicts, not a design choice.

Adding a New Algorithm Today
  1. Create a new directory: MCP_new_algo/
  2. Create a new Python venv: python -m venv venv
  3. Install all dependencies into that venv
  4. Write a new server.py MCP server (~200 lines of boilerplate)
  5. Add the MCP tool definitions (input schema, handler, output)
  6. Write the training logic inside the new server
  7. Update the recommender's get_mcp_paths() with hardcoded Windows path
  8. Add scoring rules in score_algorithm() for the new algo
  9. Test the new MCP server independently
  10. Test the recommender → new MCP handoff

~10 steps, ~300+ lines of new code, a new venv, a new subprocess to manage.

Adding a New Algorithm in Our Architecture
  1. Add one line to the model registry dict in train_model.py:
MODELS = {
  # existing algorithms...
  "xgboost": XGBClassifier,
  "random_forest": RandomForestClassifier,
  "lightgbm": LGBMClassifier,
  "logistic_regression": LogisticRegression,
  "catboost": CatBoostClassifier,  # NEW
}
  1. Add scoring rules in score_algorithms.py (~20 lines)
  2. Add catboost to the Dockerfile deps (one line)
  3. Done. Same container, same subagent, same workflow.

~4 steps, ~25 lines changed, zero new infrastructure.

✓
The core insight: Training is a generic pattern -- load, preprocess, fit, evaluate, register. The only variable is the model class. One script with a dict mapping handles all current and future algorithms. Testing a new model type is as fast as adding a key to the dict and rerunning.

2 Our Architecture: Agentic, Not Scripted

The key shift: the LLM is in the loop making decisions, not just executing a fixed pipeline. Scripts are tools the agent can call, not a hardcoded sequence. The agent reasons about results, adapts when things go wrong, and naturally extends to new model types without code changes.

Fixed Pipeline vs. Agentic

Fixed Pipeline (what hardcoded scripts give you)
  • Always runs the same 7 profiling checks
  • Always uses the same hand-tuned scoring rules
  • Always trains with default hyperparameters
  • If the model performs poorly, a human investigates
  • Adding a new model type requires code changes
  • Can't handle unexpected data issues (high nulls, weird distributions)

This is just rebuilding what Lisa has, but in Snowflake.

Agentic (what Cortex Code gives you)
  • Agent looks at profiling results and reasons: "high class imbalance + non-linear patterns → weight tree models higher"
  • If model underperforms, agent adapts: "F1 is 0.62, let me try tuning hyperparameters or the #2 recommendation"
  • Agent flags data issues: "47% missing in column X -- should we impute or drop?"
  • As new models appear in snowflake.ml, agent naturally supports them without code changes
  • Agent can explain its reasoning in plain English to the non-technical user
  • Agent can iterate: profile → recommend → train → evaluate → "not good enough" → try again with different approach

This is what makes it an agent, not a pipeline.

3 The 5 Subagents

Each subagent is a general-purpose child agent launched via the Task tool. It gets a prompt, has access to tools, runs a Python script, and returns JSON to the orchestrator. Scripts run on SPCS.

1
Data Profiler
Subagent
Handles CSV upload or Snowflake table input. Profiles the data: schema, row count, linearity, FFT patterns, class balance, feature types, multicollinearity. Direct port of Lisa's analyze_csv() + analyze_fft_patterns().
What this subagent does
  • If CSV: upload to Snowflake stage, COPY INTO a table
  • If table: validate it exists, DESCRIBE schema
  • Sample data, compute row count, null %, class balance
  • Pearson correlations per feature vs target
  • FFT separability analysis (pass vs fail frequency patterns)
  • Perceptron convergence test for linear separability
  • Correlation matrix for multicollinearity check
Tools & Libraries
sql_execute pandas numpy scipy.stats scipy.fft sklearn SPCS Container
Passes to Subagent 2
  • JSON profile: n_samples, n_features, avg_linearity, linearly_separable, fft_analysis, class_balance, high_correlation, missing_percent, numeric_features, categorical_features
2
Algorithm Recommender
Subagent
Scores 9 algorithms against the profile. Returns top 3 with scores, reasoning, strengths, and weaknesses. Lisa's AlgorithmRecommender class (~560 lines) lifts directly into this subagent's script.
What this subagent does
  • Receives profile JSON from Subagent 1
  • Runs score_algorithms.py -- portable scoring engine
  • Scores each of 9 algorithms with weighted heuristics
  • Factors: linearity, sample size, feature types, FFT strength, class balance, multicollinearity
  • Ranks by composite score, returns top 3
Tools & Libraries
bash (uv run) Pure Python
🔎 No Snowflake calls. This subagent runs purely on the profile dict. No SQL, no Snowpark. Lisa's code lifts as-is.
Passes to Subagent 3
  • Top algorithm key (e.g. "xgboost")
  • Full recommendation report for user
3
Model Trainer
Subagent
Trains the recommended algorithm on the Snowflake table using snowflake.ml. Handles preprocessing (encoding, scaling), train/test split, model fitting, and evaluation. Replaces all 8 MCP training servers.
What this subagent does
  • Creates Snowpark DataFrame from the table
  • Auto-encodes categoricals (OrdinalEncoder / OneHotEncoder)
  • Train/test split (configurable ratio, default 80/20)
  • Selects correct snowflake.ml.modeling class based on algo key
  • Fits model on training data
  • Evaluates: accuracy, F1, precision, recall, AUC, confusion matrix
Algorithm → snowflake.ml Mapping
  • xgboost → XGBClassifier
  • random_forest → RandomForestClassifier
  • lightgbm → LGBMClassifier
  • logistic_regression → LogisticRegression
  • naive_bayes → GaussianNB
  • svm_linear / svm_rbf → SVC
Tools & Libraries
snowflake.ml Snowpark Python sql_execute SPCS Container
Passes to Subagents 4 & 5
  • Trained model object + metrics dict
  • Feature names used in training
4
Feature Reporter
Subagent
Extracts feature importances from the trained model. SHAP values for tree models, coefficients for linear models. Returns a ranked list of which features most drive pass/fail. This is the actionable "why" output.
What this subagent does
  • Loads trained model from Subagent 3
  • Tree models: extract feature_importances_
  • Optionally compute SHAP values (TreeExplainer)
  • Linear models: extract coefficients
  • Rank all features by absolute importance
  • Note direction of correlation (+/-) per feature
  • Generate plain-English summary for non-technical users
Tools & Libraries
Snowpark Python shap numpy SPCS Container
Output to User
  • Top N features ranked by importance with scores
  • Direction of impact per feature
  • Summary: "The top driver of test failure is X with Y% importance"

4 Why Separate Subagents? Best Practices

Each subagent is its own .cortex/agents/*.md file with a focused system prompt, scoped tools, and its own context window. This is how production Cortex Code agents are built.

Anti-Pattern: One Big SKILL.md
  • All 5 steps defined inline in one file
  • Every step shares the same context window
  • Profiler's 5000 rows of FFT data pollute the trainer's context
  • All tools available to all steps (no scoping)
  • Harder to test, debug, or reuse individual steps
  • Context fills up fast — hits compaction before training finishes
Best Practice: Separate .cortex/agents/ Files
  • Each agent has its own markdown file with focused system prompt
  • Each runs in its own context window — returns only a summary
  • Profiler returns JSON profile, trainer never sees raw FFT data
  • Tools scoped per agent (recommender is read-only, trainer gets write)
  • Each agent independently testable: "Use the ml-data-profiler on MY_TABLE"
  • CoCo auto-discovers agents when you cd into the repo

5 Subagent Best Practices

1. Single Responsibility

One agent = one job. The profiler profiles. The recommender recommends. The trainer trains. Clear input, clear output, clear handoff.

2. Context Preservation

Each subagent runs in its own context window. Profiling 5000 rows of FFT data never floods the training step. Returns a summary, not raw output.

3. Permission Hygiene

Scope tools per agent. Recommender: read-only. Trainer: write + SQL. Registrar: SQL + registry only. Minimizes blast radius of errors.

4. Cost Control via Model Routing

Route simple tasks to Haiku (fast, cheap). Route complex tasks to Sonnet. The algorithm recommender is pure Python scoring — Haiku handles it in seconds. The trainer needs reasoning about metrics — Sonnet.

5. Auto-Delegation via Description

CoCo reads each agent's description field to decide when to delegate. Write action-oriented descriptions: "Profile a Snowflake table for ML readiness, computing linearity, FFT patterns, and class balance."

Tool Scoping Per Subagent

Subagent Tools Allowed Model Why
Data Profiler Bash sql_execute Read Sonnet Needs SQL + heavy compute for FFT analysis
Algorithm Recommender Bash Read Haiku Pure Python scoring — fast and cheap
Model Trainer Bash sql_execute Read Write Sonnet Needs snowflake.ml + write access + reasoning about metrics
Feature Reporter Bash sql_execute Read Sonnet Needs SHAP computation + data access
Model Registrar Bash sql_execute Haiku Registry calls only — fast and cheap

How Auto-Delegation Works

Delegation Walkthrough
User: Analyze MY_TABLE targeting PASS_FAIL and train the best model

CoCo: Reads SKILL.md — this matches the ml-auto-recommender workflow
SKILL.md says: first, profile the data

▶ Delegates to ml-data-profiler (own context, Sonnet, scoped tools)
  Profiler runs profile_data.py, computes FFT/linearity/class balance
  Returns JSON summary — main context never sees 5000 rows of raw data

▶ Delegates to ml-algo-recommender (own context, Haiku, read-only)
  Recommender runs score_algorithms.py on the profile JSON
  Returns top 3 — agent explains reasoning to user, asks to proceed

▶ Delegates to ml-model-trainer (own context, Sonnet, full write access)
  Trainer runs train_model.py, notices class imbalance, adds class_weight='balanced'
  Returns metrics — agent evaluates, may retry if F1 < 0.6

▶ Delegates to ml-feature-reporter + ml-model-registrar
  Feature report + model registration happen, results returned to user

Each step: own context → scoped tools → summary back to orchestrator

Key Files Explained

SKILL.md — Lightweight orchestrator. Defines the workflow and when to delegate to each agent. No inline subagent prompts — those live in the agent files.

.cortex/agents/*.md — 5 subagent definitions. Each has a name, description, scoped tools list, optional model override, and a focused system prompt. CoCo auto-discovers these from the project directory.

scripts/*.py — The tools the agents call. Each is self-contained with argparse CLI args. Run via uv run or directly in the SPCS container.

Dockerfile — Builds the SPCS image with all deps: scipy, shap, xgboost, lightgbm, snowflake-ml-python.

setup/*.sql — One-time Snowflake setup: database, image repository, compute pool, SPCS service.
